handler机制，四个组成部分及源码解析

四个组成部分

  • Handler
  • Looper
  • Message
  • MessageQueue

简单回答一下。handler用来帮助android处理线程间通信。

每个handler创建时，需要Looper，每个Looper都会有MessageQueue。

handler在sendMessage时，会将自己赋值给message.target，然后将message添加之 MessageQueue的单向链表，链表根据指定时间排序。

其中Looper的死循环会一直调用MessageQueue的next方法，直到MessageQueue返回了message对象，然后通过message中的target调用handleMessage。正是因为Looper的死循环“捡起”message对象，这个操作，确保了handleMessage方法总是在Looper被创建的线程被回调的

而MessageQueue的next方法是通过单向链表中的message的保存的回调时间与当前之间对比，如果满足条件则返回，否则阻塞时间差值

（当然讲到looper肯定又会讲到ThreadLoacl，这个留给后面部分）

<merge> ：如果被inclued的根布局相关属性和使用的布局相同则可以使用减少减少布局inclued时嵌套层次

<viewstub>：用作占位，只有被主动调用inflate 方法或者setVisibility方法时才会加载布局。

实现原理不是很了解，但是<merge>肯定是 LayoutInflate时做了相关处理(对标签是<merge>时，向上一层传递属性吧)。

  • JVM垃圾回收策略，怎样判对象、类需要被回收？

    可达性分析，如果一个对象没有被静态变量，natvie方法，或者当前栈中强引用，则说明这个对象是可以被回收的

  • 四种垃圾回收算法标记-清除、复制、标记-整理、分代收集。

    • 标记-清除， GC时如果发现一个对象是可以被回收的则进行清除，缺点是内存容易碎片化
    • 复制，将内存空间分成两份，GC时将一份内存中所有存活的对象拷贝到另外一份内存中，缺点是内存只剩下一半，优点是内存排列整齐
    • 标记-整理，GC时发现对象可以被回收则进行清除，不可以回收则进行排列。
    • 分代收集，将内存分为新生代，老年代，和永久代。新生代的占用空间比例较小，使用复制算法，老年代使用标记整理算法。如果新生代对象每次经历过GC都会将对象头上的年龄增加，一定次数后会升级至老年代或者幸存区被填满剩余对象也会进入老年区。（大对象也会进入老年区）
