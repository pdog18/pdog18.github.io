一个简单的基础库、架构

一个新项目每次都要处理一些新的基础库，整理一下我使用的方式。

项目结构

  • foundation
    • core
    • api
    • resource
  • app
  • module…

foundation 依赖于 core、api、resource

app依赖于foundation

module依赖于foundation

Module 大致功能介绍
foundation 基础库，这里统一引用了三个模块，然后注册了一个的RxJava 钩子
core 扩展方法、核心工具类、Base合集、大部分的功能性依赖
api 网络工具类、自定义网络异常、异常相关处理、数据模型
resource 统一的资源（图片，文字，颜色）
app 完全空的一个module，作为一个壳
module… 依赖于foundation 编写每个模块

架构简单介绍

传统的MVP架构会对View和Presenter都向上抽取接口。

而本项目中Presenter没有向上抽取接口，原因是普通项目中Presenter抽取接口意义不大，规范太严格，代码容易很死板。

BaseView.kt中只是简单规定了需要持有一个P类型的对象，和一个compositeDisposable，并且在继承时必须使用lazy{ } 来初始化，防止每次调用都获取新的对象。

而在BaseActivity.kt （继承于ViewContainer ）中的onCreate方法中依次调用了3个方法。

实际运行时调用方法顺序如下

  1. doBeforeInflateContextView()
  2. initViews()
  3. loadData()

使用

编写新的界面时

  • View层
    1. 创建对应的View层接口例如GankView 继承于BaseView ，声明各种接口方法
    2. 创建新的Activity 继承 BaseActivity，然后根据界面是否需要Mvp 模式
      • MVP -> 实现对应的接口，（如GankView ），并且使用泛型声明指定的Presenter
      • 普通 -> 不需要实现任何接口
  • Presenter层
    1. 没有任何规定，持有View的时候记得使用接口形式持有
  • 网络请求
    1. 创建对应的Service接口，及对应的接口返回实体
    2. 参考retrofit使用
